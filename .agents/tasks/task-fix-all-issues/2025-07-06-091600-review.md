# v2.0.0 major feature drop: status lifecycle, vendor responses, upvotes, admin hub, delete actions

This release transforms the Community Board from a simple submit-and-approve flow into a multi-stage request lifecycle with vendor engagement. Six features land together: non-destructive uninstall + housekeeping fixes, a status workflow (In Progress / Fulfilled) with vendor responses and upvotes, search/filter/duplicate-detection on the public board, a branded admin hub, configurable image upload count, and manual delete functionality.

**Watch for:** XSS via unescaped match data in duplicate-detection JS output (confirmed). Hub dashboard counts query `_zcrb_status` meta but statuses are registered as custom post statuses — the counts will always return 0 (confirmed). The `image_max_count` setting is stored and displayed in the admin UI but never enforced in the form template or AJAX handler (confirmed). No confirmation on the single-item delete row action — any admin who clicks it permanently destroys data with one click (confirmed).

## High-level view

The status lifecycle registers two custom WordPress post statuses (`zcrb_in_progress`, `zcrb_fulfilled`) via `register_post_status()`. However, the admin hub dashboard queries `_zcrb_status` meta instead of using the `post_status` field, so the "In Progress" and "Fulfilled" stat cards will perpetually show 0.

The vendor response system fires a custom action (`zcrb_vendor_responded`) that the notification class hooks into cleanly. The structural gap is in the JS layer: match data from the duplicate-detection endpoint is inserted via `innerHTML` without escaping, creating an XSS vector.

The duplicate-detection feature triggers on 10+ characters with a 500ms debounce, queries published posts via `WP_Query` `s` parameter, and renders up to 3 matches. This is basic keyword matching, not semantic similarity.

The delete path calls `ZCRB_Retention::delete_request()` which permanently removes posts and their attachments — there is no soft-delete or confirmation dialog for the single-row action.

The `image_max_count` setting (1-4 images per submission) has a UI control and sanitization, but the form still renders a single `<input type="file">` and the AJAX handler doesn't enforce the limit. This is a dead setting until the frontend and backend are updated.

<summary>Issues (7)</summary>

1. **XSS in duplicate-detection rendering** — `initDuplicateDetection` injects `match.title` and `match.permalink` into the DOM via `innerHTML` without escaping. Sanitize or use `textContent`/`createElement` for user-generated content.
2. **Hub stats always zero for custom statuses** — `ZCRB_Admin_Hub::count_posts_by_meta('_zcrb_status', ...)` queries a meta key that is never written. The statuses are stored in `post_status` column. Query `wp_count_posts()` or use `post_status` in `WP_Query`.
3. **image_max_count setting is unenforced** — The form template still renders one file input and the AJAX handler doesn't validate against the limit. Either enforce it or remove the setting until multi-upload is implemented.
4. **No confirmation on permanent delete row action** — A single click on "Delete Permanently" in the list table immediately and irrevocably removes the post. Add a JS `confirm()` prompt or use WordPress trash flow.
5. **Upvote race condition under concurrency** — `get_post_meta` → modify array → `update_post_meta` is not atomic. Two simultaneous upvotes can lose one write. Low risk at expected scale but worth noting.
6. **Vendor response stored with `current_time('timestamp')`** — This function is deprecated since WP 5.3. Use `time()` or `current_datetime()->getTimestamp()`.
7. **Notification email lacks HTML content type** — `wp_mail()` sends plain text with `{link}` as a raw URL. Consider setting `Content-Type: text/html` or at minimum ensuring the URL is on its own line for email client auto-linking.

## XSS in duplicate-detection innerHTML

In `zcrb.js`, the `initDuplicateDetection` function builds HTML from AJAX response data and injects it via `innerHTML`:

```javascript
html += '<a class="zcrb-similar__item" href="' + match.permalink + '" target="_blank" rel="noopener">';
html += '<span class="zcrb-similar__item-ref">#' + match.ref + '</span>';
html += '<span class="zcrb-similar__item-title">' + match.title + '</span>';
html += '</a>';
```

The `match.title` value comes from `wp_trim_words($post->post_content, 12, '...')` on the server, which strips tags but does not HTML-entity-encode the output. If post content contains angle brackets that survive `wp_trim_words`' word-splitting logic, the resulting string could execute as markup. More critically, `match.permalink` is a URL that could theoretically be crafted to include `javascript:` protocol if permalink structures are manipulated.

The vendor response JS has the same pattern but is gated behind `canVendorRespond` (editors only), reducing the attack surface. The duplicate-detection panel is shown to all logged-in users.

Fix: use `document.createElement` and `textContent` assignments, or run a simple HTML-entity-escape function on the values before string concatenation.

## Hub dashboard counting the wrong thing

`ZCRB_Admin_Hub::render_page()` computes stats:

```php
$in_progress = $this->count_posts_by_meta( '_zcrb_status', 'in_progress' );
$fulfilled   = $this->count_posts_by_meta( '_zcrb_status', 'fulfilled' );
```

But `ZCRB_Status::register_statuses()` registers `zcrb_in_progress` and `zcrb_fulfilled` as WordPress post statuses stored in `wp_posts.post_status`. Nothing in the codebase ever writes a `_zcrb_status` meta key. These counts will always be 0.

The fix:

```php
$in_progress = $this->count_posts( array( 'zcrb_in_progress' ) );
$fulfilled   = $this->count_posts( array( 'zcrb_fulfilled' ) );
```

## image_max_count: setting without enforcement

The settings page shows a dropdown for 1-4 images and the sanitizer clamps the value. But `render_form()` in `class-zcrb-template.php` still renders a single `<input type="file" name="zcrb_image">`. The localized JS config doesn't include `imageMaxCount`. The AJAX submission handler in `class-zcrb-form.php` (unchanged from v1.4.4) processes `$_FILES['zcrb_image']` as a single upload.

The setting is inert. A user can only upload one image regardless of the configured maximum.

## Permanent delete with no safety net

The "Delete Permanently" row action in `class-zcrb-admin.php` calls `ZCRB_Retention::delete_request($post_id)` which force-deletes the post, its meta, and attached media. The link is a standard `<a href>` — no JavaScript confirmation, no intermediate trash state. WordPress's built-in "Delete Permanently" requires the post to be in Trash first, providing a two-step safety net. This custom implementation bypasses that entirely.

The bulk delete action has the same behavior but WordPress bulk actions at least require selecting checkboxes and choosing from a dropdown, which provides some friction. The single row action is one accidental click away from permanent data loss.

## Upvote concurrency and deprecated timestamp

The upvote toggle in `ZCRB_Upvote::ajax_upvote()` does a read-modify-write cycle on the `_zcrb_upvoted_users` meta array without any locking. Two near-simultaneous requests could both read the same array state, each apply their modification, and the second `update_post_meta` call overwrites the first. At typical WordPress single-server scale this is unlikely to cause visible data loss, but for a popular request with many upvotes arriving at once, votes can be dropped.

Both `ZCRB_Vendor_Response::add_response()` and the upvote handler use `current_time('timestamp')` which has been deprecated since WordPress 5.3 (the plugin declares `Requires at least: 5.8`). It still works but triggers deprecation notices on debug-enabled sites.
